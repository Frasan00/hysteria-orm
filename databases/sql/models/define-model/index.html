Programmatic Models (defineModel)

Hysteria ORM provides two ways to define models:

  1. Decorator-based (class syntax) — covered in Defining SQL Models
  2. Programmatic (defineModel function) — covered here

Both approaches produce identical Model subclasses that work seamlessly with queries, relations, hooks, migrations (syncSchema), and everything else in the ORM. Choose whichever style fits your project.

defineModel is particularly useful when:

  • You prefer a functional style over decorators
  • You're working in plain JavaScript (no decorator transpilation needed)
  • You want to generate models dynamically at runtime

Quick Start​

import { defineModel, col, rel } from "hysteria-orm";

const Post = defineModel("posts", {
  columns: {
    id: col.increment(),
    title: col.string(),
    body: col.text(),
    userId: col.integer(),
    createdAt: col.datetime({ autoCreate: true }),
    updatedAt: col.datetime({ autoCreate: true, autoUpdate: true }),
  },
});

const User = defineModel("users", {
  columns: {
    id: col.increment(),
    name: col.string(),
    email: col.string({ nullable: false }),
    isActive: col.boolean(),
    createdAt: col.datetime({ autoCreate: true }),
    updatedAt: col.datetime({ autoCreate: true, autoUpdate: true }),
  },
  relations: {
    posts: rel.hasMany(() => Post, "userId"),
  },
  indexes: [["email"]],
  uniques: [["email"]],
  hooks: {
    beforeFetch(qb) {
      qb.whereNull("deleted_at");
    },
  },
});

The returned User and Post are fully-typed Model constructors you can use exactly like decorator-based models:

const users = await User.query().where("isActive", true).many();
const user = await User.insert({ name: "Alice", email: "alice@example.com" });

API Reference​

defineModel(table, definition)​

ParameterTypeDescription
tablestringThe database table name
definitionModelDefinitionObject describing columns, relations, etc.

Returns a concrete Model subclass.

ModelDefinition​

KeyTypeRequiredDescription
columnsRecord<string, ColumnDef>YesColumn definitions using the col namespace
relationsRecord<string, RelationDef>NoRelation definitions using the rel namespace
indexesIndexDefinition[]NoTable indexes
uniquesUniqueDefinition[]NoUnique constraints
checksCheckDefinition[]NoCheck constraints
hooksHooksDefinitionNoLifecycle hooks
optionsDefineModelOptionsNoCase conventions, soft delete, etc.

Column Descriptors (col)​

The col namespace mirrors all @column decorators. Each method returns a ColumnDef that carries the correct TypeScript type for full intellisense.

MethodInferred TypeDescription
col()anyStandard column
col.primary()anyShorthand for col({ primaryKey: true })
col.increment()number | null | undefinedAuto-incrementing integer primary key
col.bigIncrement()number | null | undefinedAuto-incrementing bigint primary key
col.integer()number | null | undefinedInteger column
col.bigInteger()number | bigint | null | undefinedBig integer column
col.float()number | null | undefinedFloat column
col.decimal()number | null | undefinedDecimal column with optional precision and scale
col.string()string | null | undefinedVARCHAR column with optional length
col.text()string | null | undefinedLONGTEXT column
col.boolean()boolean | null | undefinedBoolean column
col.json()unknownJSON/JSONB column (auto serialize/parse)
col.date()Date | string | null | undefinedDATE column (YYYY-MM-DD)
col.datetime()Date | string | null | undefinedDATETIME column (YYYY-MM-DD HH:mm:ss)
col.timestamp()Date | string | null | undefinedUnix timestamp column
col.time()Date | string | null | undefinedTIME column (HH:mm:ss)
col.uuid()string | null | undefinedAuto-generates UUID if not provided
col.ulid()string | null | undefinedAuto-generates ULID if not provided
col.binary()Buffer | Uint8Array | string | null | undefinedBinary/blob column
col.enum(values)values[number] | null | undefinedEnum constrained to the given values array
col.encryption.symmetric(opts)string | null | undefinedSymmetric encryption column
col.encryption.asymmetric(opts)string | null | undefinedAsymmetric encryption column

Each method accepts the same options as its decorator counterpart (e.g., nullable, hidden, primaryKey, autoCreate, autoUpdate, databaseName, default).

const Product = defineModel("products", {
  columns: {
    id: col.increment(),
    name: col.string({ nullable: false }),
    price: col.decimal({ precision: 10, scale: 2 }),
    status: col.enum(["draft", "published", "archived"] as const),
    metadata: col.json(),
    createdAt: col.datetime({ autoCreate: true }),
  },
});

Relation Descriptors (rel)​

The rel namespace mirrors the @hasOne, @hasMany, @belongsTo, and @manyToMany decorators.

MethodInferred TypeDescription
rel.hasOne(model, foreignKey?)InstanceType<M> | null | undefinedOne-to-one (foreign key on the other model)
rel.hasMany(model, foreignKey?)InstanceType<M>[] | null | undefinedOne-to-many
rel.belongsTo(model, foreignKey?, options?)InstanceType<M> | null | undefinedInverse of hasOne/hasMany
rel.manyToMany(model, through, keys?, options?)InstanceType<M>[] | null | undefinedMany-to-many via join table

The model parameter is always a lazy callback (() => ModelClass) to support circular references.

const Profile = defineModel("profiles", {
  columns: {
    id: col.increment(),
    bio: col.text(),
    userId: col.integer(),
  },
  relations: {
    user: rel.belongsTo(() => User, "userId"),
  },
});

const User = defineModel("users", {
  columns: {
    id: col.increment(),
    name: col.string(),
  },
  relations: {
    profile: rel.hasOne(() => Profile, "userId"),
    posts: rel.hasMany(() => Post, "userId"),
    roles: rel.manyToMany(() => Role, "user_roles", {
      leftForeignKey: "userId",
      rightForeignKey: "roleId",
    }),
  },
});

rel.belongsTo Options​

The optional third argument accepts constraint options:

OptionTypeDescription
onDeleteOnUpdateOrDeleteForeign key ON DELETE action
onUpdateOnUpdateOrDeleteForeign key ON UPDATE action
constraintNamestringCustom constraint name

rel.manyToMany Keys​

When the through table is passed as a string, provide key mappings:

OptionTypeDescription
leftForeignKeystringColumn in the join table referencing this model
rightForeignKeystringColumn in the join table referencing the related model

Indexes, Uniques & Checks​

Constraints are defined as arrays, matching the behavior of @index, @unique, and @check decorators.

Indexes​

defineModel("users", {
  columns: { /* ... */ },
  indexes: [
    ["email"],                                        // array form
    { columns: ["name", "email"], name: "idx_name_email" },  // object form with custom name
  ],
});

Uniques​

defineModel("users", {
  columns: { /* ... */ },
  uniques: [
    ["email"],
    { columns: ["email"], name: "uq_users_email" },
  ],
});

Checks​

defineModel("users", {
  columns: { /* ... */ },
  checks: [
    "age >= 18",                                           // string form
    { expression: "status IN ('active', 'inactive')", name: "chk_status" },  // object form
  ],
});

Hooks​

Lifecycle hooks let you run logic before or after certain operations. They work identically to static hook methods on decorator-based models.

HookSignatureWhen it runs
beforeFetch(queryBuilder: ModelQueryBuilder) => void | Promise<void>Before any SELECT query
afterFetch(data: any[]) => any[] | Promise<any[]>After SELECT, can transform results
beforeInsert(data: any) => void | Promise<void>Before a single INSERT
beforeInsertMany(data: any[]) => void | Promise<void>Before a bulk INSERT
beforeUpdate(queryBuilder: ModelQueryBuilder) => void | Promise<void>Before an UPDATE query
beforeDelete(queryBuilder: ModelQueryBuilder) => void | Promise<void>Before a DELETE query
const AuditedModel = defineModel("audited_items", {
  columns: {
    id: col.increment(),
    name: col.string(),
    createdAt: col.datetime({ autoCreate: true }),
  },
  hooks: {
    beforeInsert(data) {
      data.name = data.name?.trim();
    },
    afterFetch(rows) {
      return rows.filter((r) => r.name !== "DELETED");
    },
  },
});

Options​

OptionTypeDefaultDescription
modelCaseConventionCaseConvention"camel"Case style for model property names
databaseCaseConventionCaseConvention"snake"Case style for database column names
softDeleteColumnstring"deletedAt"Column used for soft deletes
softDeleteValueboolean | stringCurrent UTC datetimeValue written on soft delete
const LegacyModel = defineModel("legacy_records", {
  columns: {
    record_id: col.increment(),
    record_name: col.string(),
  },
  options: {
    modelCaseConvention: "snake",
    databaseCaseConvention: "snake",
    softDeleteColumn: "is_deleted",
    softDeleteValue: true,
  },
});

Complete Example​

Here is the decorator-based User model from Defining SQL Models rewritten with defineModel:

import { defineModel, col, rel } from "hysteria-orm";
import { Post } from "./Post";
import { Address, UserAddress } from "./Address";

export const User = defineModel("users", {
  columns: {
    id: col.increment(),
    name: col.string(),
    email: col.string(),
    password: col({ hidden: true }),
    status: col.enum(["active", "inactive"] as const),
    isActive: col.boolean(),
    balance: col.decimal({ precision: 10, scale: 2 }),
    metadata: col.json(),
    createdAt: col.datetime({ autoCreate: true }),
    updatedAt: col.datetime({ autoCreate: true, autoUpdate: true }),
  },
  relations: {
    post: rel.hasOne(() => Post, "userId"),
    posts: rel.hasMany(() => Post, "userId"),
    addresses: rel.manyToMany(() => Address, () => UserAddress, {
      leftForeignKey: "userId",
      rightForeignKey: "addressId",
    }),
  },
  indexes: [["email"]],
  uniques: [["email"]],
});

Both the decorator version and the defineModel version produce the same Model class. They register identical column metadata, relation metadata, and constraints — so migrations, queries, and hooks all work the same way.

TypeScript Types​

You can export the inferred instance type for use elsewhere:

import type { InferModel } from "hysteria-orm";

const userDef = {
  columns: {
    id: col.increment(),
    name: col.string(),
    email: col.string(),
  },
} as const;

export const User = defineModel("users", userDef);

// The instance type, useful for function signatures
type UserInstance = InstanceType<typeof User>;
